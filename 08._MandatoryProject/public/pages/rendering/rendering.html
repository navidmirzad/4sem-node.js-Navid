SSR

  • SEO Optimized: Because SEO primarily looks through the keywords
    on the server rendered content, hence why SSR is more search engine optimized
  • Loads incredibly fast: As the server is rendering a static page, and therefore
    is no need to wait for any async/await promises
  • No JS dependency: because it is a static website, also we can dynamically
    inject JS if needed
  • Not ideal if there is a lot of interactivity on the webpage. And then again
    We can always inject JS-interactivity ourselves, and just have some components SSR.

CSR

  • Fast site rendering after initial load: Client side rendering loads slower
    compared to SSR, as there is API response delays, and we
    have to wait until it loads the data.
  • Initial HTML is minimal: The initial HTML file contains just the structure,
    and references to the JS files. The browser then executes the js, and generates content
    on the client-side, which is the opposite to SSR.
  • Not great for SEO: As a the content is rendered on the client-side,
    it is therefore not on the initial page when the SEO scans for keywords.

Server side rendering:

  • In my app.js I have the line: 
    import express from "express";

const app = express();
    
app.use(express.static("public"));
  • First thing we do is import express from the express libary which is a great Node.JS web framework.

  • const app = express(); - is where we create an instance of express, and use it for routing and handling HTTP requests

  • app.use(express.static("public")); - is where I tell express to serve static files from my public directory.
